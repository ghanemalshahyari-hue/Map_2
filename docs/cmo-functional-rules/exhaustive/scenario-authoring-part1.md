## 1. Foundation: database, clock, environment, realism

### Database (DB) selection before scenario build
- **Models:** The platform/weapon reference dataset that defines all unit capabilities for the scenario's era. Choosing it is the canonical *first* authoring step.
- **Inputs / parameters:** Selected database file (e.g. DB3000 "for 72", a newer DB "for 80"); scenario era intent (modern vs Cold War); the chosen scenario **date** (interacts with DB to gate availability).
- **Behavior / rules:** `Editor → Database`. Must be set **BEFORE adding anything** ("critical"; "set the database before you do anything"; set it **before importing units**). The database determines which platforms/weapons exist and their stats: an older DB for a Cold-War scenario, a newer DB for modern. The chosen scenario **date** further gates which platforms are selectable — a unit not yet in service for that date won't appear. Changing the DB later is **not advised / destructive** — pick the DB first, then build.
- **Outputs / effects:** Constrains the entire pool of available units/weapons and their performance characteristics for the scenario; combined with date, fixes the era-constrained roster.
- **Edge cases / quirks:** The default "current" DB shown may differ from the newest available; exact DB version differs over time/patches. Author heuristic from one tutorial: cap scenarios at **2008** so aircraft remain usable in DCS later (any aircraft in DCS is 2008-or-earlier). Quick Battle examples specifically require the **Modern** DB (not the classic DB) — option pools differ between databases.
- **Source:** vy5glbQ1G6k; ixu2x6doLFA
- **Confidence:** High

### Scenario time, start time, and duration (with end-condition validation)
- **Models:** The simulation clock — current time, the canonical start time, and how long the scenario runs — plus the validation the editor enforces on the end condition.
- **Inputs / parameters:** Scenario (current) time [date/time]; scenario **START** time [date/time] (a *distinct* field); **Duration** (days/period); optional daylight-savings flag; implied local-vs-Zulu offset; localization affects dialog layout.
- **Behavior / rules:** `Editor → Scenario Times and Duration`. "Scenario time" (current) is **NOT** the same as "scenario start time" — both must be set consistently. **Hard constraints the editor enforces:** (1) Duration must be a **POSITIVE** number; (2) the CURRENT time must **NOT exceed start time + duration**, or the scenario **ENDS INSTANTLY on load**. The chosen date also gates platform availability (see Database). Worked example: start **05:30** local, **10-hour** duration, date **September 10, 2008**, with a noted **2-hour** local-vs-Zulu difference for the region.
- **Outputs / effects:** Defines the clock, day/night lighting, the era-constrained roster, and the scenario's end condition.
- **Edge cases / quirks:** Newer builds may **auto-copy** start time into current time when you set it; older builds did not (version-dependent). Localization can relabel/reshuffle the dialog. The current-vs-start distinction is the classic gotcha — mis-setting current beyond start+duration causes immediate end on load.
- **Source:** vy5glbQ1G6k; ixu2x6doLFA
- **Confidence:** High

- **Outputs / effects:** Scenario metadata only.
- **Edge cases / quirks:** None stated.

### Weather model (temperature, rainfall, cloud-cover tiers, wind/sea state)
- **Models:** A largely static environmental state affecting sensors, weapons, and ship performance.
- **Inputs / parameters:** Average (sea-level) temperature [°C]; rainfall rate; sky/cloud-cover setting (stepped through discrete regimes); wind/sea-state setting (stepped, Beaufort-like). Lua can vary weather over time.
- **Behavior / rules:** Weather is **static** unless scripted with Lua.
  - **Temperature:** entered as average **SEA-LEVEL** temp and is **MODIFIED BY ALTITUDE** (higher terrain ⇒ lower local temp). **Higher temperature DECREASES** heat-seeking (IR) weapon effectiveness; **lower temperature INCREASES** it.
  - **Cloud/sky setting** cycles through *grouped, non-monotonic* regimes as you step it: low (1 click) = low clouds **~5,000–7,000 ft** (small puffy, expected most days, nothing above); step up = clouds rise to **~10,000** then **~16,000 ft**; further = light high clouds **~20,000–23,000 ft**; then clouds drop back to **~2,000–7,000 ft** but get **THICKER**; further still = moderate high clouds with **~50%** cover; maximum = **thick fog 0–2,000 ft** plus solid cloud **7,000–36,000 ft** (a vertical wall) with visibility of a few feet/yards.
  - **Wind/sea state:** 0 = no wind; one notch up shown as "sea state 1" (little wind); typical ocean ops are **sea state 3–4**. Sea state affects ship performance — a ship rated max **sea state 5** driven at **sea state 6** is effectively **crippled**. Wind/sea state also degrades **weapon accuracy**; at hurricane levels hitting anything is very difficult.
- **Outputs / effects:** Modifies IR-weapon effectiveness (via temp), sensor/visual detection (via cloud/fog), weapon accuracy (via wind), and ship mobility/capability (via sea state). Global environment consumed by sensor/weapon math.
- **Edge cases / quirks:** Cloud setting is **non-monotonic** (clouds first rise, then drop-but-thicken, then build a full vertical wall); displayed temperature is altitude-adjusted; exceeding a ship's rated max sea state cripples it. Authors look up real monthly climatology for the region/month and then **simplify** — worked example: averaged high/low 28 °C and 19 °C ⇒ ~24 °C, simplified and entered as **22 °C**, no rain, clear skies, a little wind. Weather can make a loadout unavailable — a weapon marked **"clear weather only"** (e.g. some anti-radiation loadouts) is unavailable under non-clear conditions.
- **Source:** vy5glbQ1G6k; wCaE47aRaHA
- **Confidence:** Med

### Scenario realism feature toggles
- **Models:** Global switches that make engagements more or less physically realistic, scenario-wide.
- **Inputs / parameters:** Detailed gun/sensor fire control; Unlimited ammunition; Magazines unlimited at airbases; Realistic submarine communications; Effects of terrain type; Communications disruption (and others).
- **Behavior / rules:** `Editor → Scenario Features/Settings`.
  - **Detailed (gun/sensor) fire control** = a platform must actually **SEE and LOCK** the target to hit it (vs move-and-fire).
  - **Unlimited ammunition / unlimited airbase magazines** remove resupply limits and **remove the need to stock magazines** — warned as bad for players because they over-load on premium munitions (JDAMs/precision) when you wanted Mark-82s.
  - **Realistic submarine communications** = a sub must be at shallow-ish depth to communicate, making deep subs hard to command ("gets tricky").
  - **Effects of terrain type** = mountains/trees make targets harder to hit.
  - **Communications disruption** = destroying a unit's radio cuts its comms, creating IFF / blue-on-blue incidents.
  - Worked example (a one-off strike scenario): Detailed Gun Control **ON**; Unlimited Magazines **ON** (justified — one-off strike, no rearm bookkeeping); Realistic Submarine Communications **DISABLED**; Communications Disruption **LEFT ENABLED**.
- **Outputs / effects:** Changes targeting requirements, ammo economy, sub command latency, terrain-based hit difficulty, and comms-loss consequences scenario-wide.
- **Edge cases / quirks:** Unlimited ammo undermines author intent on loadouts and makes **any** loadout armable regardless of ammo-storage stock (cross-references airfield rearm). Realistic sub comms "gets tricky." Comms disruption can cause friendly-fire.
- **Source:** vy5glbQ1G6k; wCaE47aRaHA
- **Confidence:** High

## 2. Sides, posture, doctrine & ROE

### Sides and the inter-side posture matrix
- **Models:** Belligerent factions and the directional diplomatic stance each holds toward every other side.
- **Inputs / parameters:** Side name (arbitrary, renameable); per-side, per-other-side posture: **Neutral / Friendly / Unfriendly / Hostile**.
- **Behavior / rules:** Add sides via `Editor → Sides → Add` (the example builds up to 4–5 sides, e.g. Syria / CIS / NATO). Posture is set **directionally**: select a side, then set how it feels about another side.
  - **Friendly** = shares intelligence information.
  - **Unfriendly** = will **not** necessarily fire but will follow/shadow you.
  - **Hostile** = will fire on sight.
  - Posture is **asymmetric**: setting Side A hostile to B does **NOT** auto-set B hostile to A — you must set both directions for symmetric hostility from the start. **However**, if one side **ATTACKS** another, the attacked side's posture **auto-flips to hostile** toward the attacker (e.g. Colombia attacks Ecuador ⇒ Ecuador auto-set hostile). The designer decides the situation ("we decide this is a hostile situation"), and posture works together with ROE/identification doctrine to drive engagements.
- **Outputs / effects:** Determines firing behavior, intel sharing, and shadowing between sides; the hostility matrix that (with doctrine) governs whether engagements occur.
- **Edge cases / quirks:** Combat automatically escalates posture on the victim side, but only after the first shot — easy to forget the reciprocal direction at authoring time. Independent of, but interacting with, Collective Responsibility.
- **Source:** vy5glbQ1G6k; ixu2x6doLFA
- **Confidence:** High

### Side briefing (rich-text / HTML)
- **Models:** The narrative/mission briefing presented to the player controlling a side.
- **Inputs / parameters:** Free text, images, effects; optional **raw HTML** edit mode for advanced formatting.
- **Behavior / rules:** Per-side briefing editor supports copy-paste, images, effects, and an HTML edit mode. (In the canonical build order this is typically done last as a polish step.)
- **Outputs / effects:** Briefing content shown to the player controlling that side.
- **Edge cases / quirks:** None stated beyond "go nuts."
- **Source:** vy5glbQ1G6k
- **Confidence:** Med

### Doctrine / Rules of Engagement with per-rule player-edit lock
- **Models:** Side- (and unit-/group-) level engagement and behavior policy, with author control over which rules the player may later change. Doctrine cascades from side → group → unit unless overridden.
- **Inputs / parameters:** Per-rule **value** PLUS an adjacent per-rule **lock checkbox** ("player can edit this rule later"). Rules called out across tutorials include: use of **nuclear weapons**; **Ignore Plotted Course When Attacking**; engagement authority / **engage opportunity targets** ("fire at everybody" / "fire at nobody"); ambiguous-target identification (**Optimistic vs Pessimistic**); air-operations tempo (**Surge vs Sustained**); **Quick Turnaround**; **fuel state** / RTB threshold; **weapons state** / Bingo behavior; ground/air-to-ground **strafing**; **jettison ordnance** behavior; **BVR** engagement logic; **automatic evasion**; SAM/WCS (Sampson/SAM mission control); ECM/EMCON; refueling/UNREP allowance; crank-and-drag.
- **Behavior / rules:** Each ROE rule has a value and a lock checkbox; if the checkbox (lock) is **ON the PLAYER CAN edit** that rule later, if **OFF it is locked**. The lock toggle is the key scenario-design lever.
  - Author guidance: **lock nuclear-weapon use OFF** so players can't use nukes you didn't intend (if you leave nuclear use editable on a nuke-carrying unit, expect the player to use them).
  - **"Ignore plotted course when attacking" = Yes** makes AI more aggressive/simpler.
  - If you **lock the AI's ability to engage to "nobody,"** units will still **FIRE IN SELF-DEFENSE** but will **NEVER initiate** an attack.
  - **Surge** = aircraft get airborne quickly but aren't sustainable (real-world surge is short-duration only); **Sustained** = slower turnaround but maintainable (modeled as rested-crew / quick-turnaround framing; the realistic default for most missions).
  - **Ambiguous-target ID:** set **Pessimistic** when ammunition is limited (don't waste shots) vs **Optimistic** to expend freely.
  - **Opportunity targets OFF** prevents attacking everything seen.
  - **Weapons-state** can be set so aircraft **disengage / leave the flight** when out of mission weapons; **jettison** allowed but paired with a leave-flight rule so empty aircraft don't loiter.
  - **BVR engagement logic is editable by the player and CANNOT be locked**, so changing it has limited author value.
- **Outputs / effects:** Governs autonomous engagement aggressiveness, evasion, fuel/weapon RTB, emissions, weapon-usage permissions, and air-cycle rates; the lock state governs player editability in-game. Stored as per-side doctrine plus a player-editability mask.
- **Edge cases / quirks:** Self-defense fire persists even when engagement authority is locked off. **BVR logic specifically cannot be locked.** Leaving Tomahawk / long-range-SAM ROE editable lets players trivialize the scenario. Crank-and-drag only matters if the platform has draggable (e.g. command-guided) weapons — moot otherwise. Jettison without a leave-flight rule wastes large amounts of ordnance. Many specific effects are qualitative.
- **Source:** vy5glbQ1G6k; ixu2x6doLFA; cMYv6wzbvmI
- **Confidence:** High

### Side flags: scenario-entry, collective response, civilian auto-tracking, preferred side
- **Models:** Side-level switches controlling player selectability/default, retaliation scope, and auto-tracking of commercial traffic.
- **Inputs / parameters:** "Player cannot select this side at start" / **Computer-Only** flag; a **Preferred Side** designation (loads by default); **Collective Response / Collective Responsibility** flag; **Auto-track civilians** (auto-track anything tagged "commercial") flag.
- **Behavior / rules:**
  - **Computer-Only / not-player-selectable:** when ON, the side is **NOT offered** to the player at scenario start (use for AI/neutral/collective sides). The non-computer-only side(s) are playable, and one is set as the **Preferred Side** that loads by default. Worked example: CIS and Syria set computer-only; NATO playable and preferred.
  - **Collective Response/Responsibility:** if **ON**, an accidental attack by one unit causes the victim side to declare war on the **ENTIRE attacking side**; if **OFF**, the victim only declares war on the single offending unit. Qualitative warning: "blow up one of them, all of them turn on me" — "can cause very interesting problems," use cautiously.
  - **Auto-track civilians:** automatically tracks any platform whose type contains **"commercial,"** useful to avoid engaging civilian shipping. Adding extra ambient sides (neutral airliners) is great realism but **significantly increases scenario complexity/effort**.
- **Outputs / effects:** Controls side availability to players, the default loaded side, breadth of war-declaration on incidents, and automatic contact tracking of civilians.
- **Edge cases / quirks:** Collective ON dramatically widens conflict from a single friendly-fire/accident; auto-track specifically keys off the **"commercial"** tag. Independent of (but interacts with) the posture matrix. More sides ⇒ more complexity/effort.
- **Source:** vy5glbQ1G6k; ixu2x6doLFA
- **Confidence:** High

### Side awareness / detection-knowledge level
- **Models:** How much a side automatically knows about contacts it detects — a scenario-wide IFF/intel realism setting.
- **Inputs / parameters:** One of: **Blind**, **Normal**, **Auto Side ID**, **(Auto Side + Unit ID)**, **Omniscient**.
- **Behavior / rules:**
  - **Blind** = eliminates detection calculations entirely (recommended for **civilian shipping** to save computation).
  - **Normal** = standard; a new detection is **Unknown** until classified.
  - **Auto Side ID** = any contact picked up is instantly known to belong to a side (simulates perfect IFF of allegiance).
  - The next level also identifies the specific **unit type**.
  - **Omniscient** = "God mode," sees everything at all times.
- **Outputs / effects:** Determines initial classification certainty of contacts and detection-processing load for that side.
- **Edge cases / quirks:** **Blind** is advised specifically for civilian sides to reduce simulation cost; **Omniscient / Auto-ID** bypass/trivialize sensing.
- **Source:** vy5glbQ1G6k
- **Confidence:** High

### Side proficiency level (stamped per-unit at placement)
- **Models:** Crew skill/experience that scales weapon-system effectiveness and survivability. Fully designer-controlled ("our universe, not the real universe") and can be set arbitrarily.
- **Inputs / parameters:** A proficiency slider with discrete tiers: **Novice, Cadet, Trained/Regular, Veteran, Ace** (transcripts name Novice / Cadet / Regular / Veteran / Trained / Ace; ordering low→high).
- **Behavior / rules:** The slider sets the proficiency that newly **PLACED** units inherit **at the moment of placement** — it is a *stamp applied as you drop units*, not a retroactive side property. You can set it to **Ace**, place units (they become aces), then lower it to **Novice** and place more (those become novices) — yielding **mixed proficiencies** in one side. Qualitative meaning: **Regular** = competent with weapon systems (or "worn out but standard"); **Trained** = a little experience; **Novice** = no experience / poorly maintained / poorly trained (worst); **Veteran** = a regular with extra practice; **Ace** = very hard to shoot down (but still destroyable by all weapons). Worked examples: Russia/CIS = Regular ("worn out"); Syria reduced to **Cadet** (less detailed training).
- **Outputs / effects:** Sets each placed unit's crew skill, affecting engagement effectiveness and how hard it is to kill.
- **Edge cases / quirks:** Proficiency is captured **at placement time**, so mixed-skill forces are built by changing the slider between placements; aces are tough but not invulnerable. Exact numeric effects not stated (qualitative); the author maps real-world maintenance/training quality onto the ladder subjectively.
- **Source:** vy5glbQ1G6k; ixu2x6doLFA
- **Confidence:** Med

## 3. Unit placement & manipulation

### Unit insertion / move / orientation
- **Models:** Placing units on the map and setting their heading.
- **Inputs / parameters:** **Insert** key + map click; unit category — **Aircraft** (incl. helos) / **Surface ship** (incl. platforms) / **Submarine** / **Facility** (everything else); name/class filter; country filter; hide-hypothetical toggle; optional custom **GUID/name**. **Move = M** key + click. **Orientation in DEGREES** via right-click → `Scenario Editor` → Set Orientation.
- **Behavior / rules:** Press **Insert**, click placement point, pick category, search by name, optionally filter by country and hide hypotheticals, optionally set a custom GUID, press OK. Placed units **CANNOT be click-dragged** — to move, select and press **M** then click destination. **Orientation is in decimal DEGREES** (not radians): **0/360 = up/north, 90 = right/east, 180 = down/south, 270 = left/west**. A placed "single" unit may actually be a **multi-vehicle group** (e.g. a "tank" is really 4 tanks). Range rings render around the unit at placement reflecting max weapon range. Proficiency is stamped from the current side slider at placement.
- **Outputs / effects:** Adds the unit at the location/heading; range rings render around it; proficiency baked in.
- **Edge cases / quirks:** The **Facility** category contains "everything"; the displayed single icon can represent multiple vehicles.
- **Source:** vy5glbQ1G6k
- **Confidence:** High

### Copy vs Clone vs Shift-Copy (cargo) and the orientation-reset quirk
- **Models:** Duplicating placed units, with rules about which payload/heading carries over.
- **Inputs / parameters:** **C** key = copy + click; **Shift+C** = copy **WITH cargo**; (Clone action via menu).
- **Behavior / rules:** **C**-key copy places a duplicate at the clicked point but does **NOT carry the original's cargo** (e.g. a C-130's howitzer cargo is dropped on copy). **Shift+C** copies the cargo as well. **QUIRK:** **cloning** a unit **RESETS its orientation back to 0** automatically (a cloned unit loses a custom heading), whereas plain copy (**C**) preserves the heading and the original retains its set heading. (This orientation reset is called out as a long-standing annoyance.)
- **Outputs / effects:** New unit instances; cargo and heading carried or reset per the rules above.
- **Edge cases / quirks:** Cargo is lost on plain copy; orientation is reset to 0 on **clone** — author must re-set heading after cloning.
- **Source:** vy5glbQ1G6k
- **Confidence:** High

### Grouping and the formation editor (relative vs fixed bearing)
- **Models:** Combining units into a led group with relative-position formation geometry, including how escorts hold station around a guarded high-value unit.
- **Inputs / parameters:** **G** = group selected; **Delete** = remove the whole group; **numpad-9** toggles normal-select mode to pick a member; **D** removes a member from a group / deletes a selected unit; **F4** = formation mode; Formation Editor (set **Group Lead**, per-member **relative position**, per-member **bearing type: Relative vs Fixed**, station distance, initial speed/heading).
- **Behavior / rules:** Select units, press **G** to group; re-selecting all members + extras and pressing **G** adds them. **Deleting a group deletes ALL members.** To break one member out: select group, press **numpad-9** to enter normal-select, pick the member, press **D**, press **9** again to return. You **CANNOT directly merge a group into another group** — CMO errors and tells you to delete-then-regroup. Formation Editor lets you reassign which unit is **Group Lead** and set each member's relative position plus a **bearing type**:
  - **Relative bearing** = the station **rotates** as the group turns (member re-orients with group turns).
  - **Fixed bearing** = the escort is pinned to an **absolute compass bearing**.
  - **Design rule:** set perimeter/threat-axis escorts to **FIXED bearing** so that even if the **player forgets to orient** the formation, the threat axis is always covered ("very very very critical"); the guarded HVU/lead and a trailing/utility unit can stay **Relative**. Worked example: escorts ~**10 nm** out on staggered bearings (front, ~20°, ~120°, south), threat-facing ones fixed.
  - When ordered to move, members queue into formation; **reversing direction keeps proper formation** (realistic).
- **Outputs / effects:** A named group with a lead and maintained relative formation during movement; a defended formation that screens the threat axis regardless of player input.
- **Edge cases / quirks:** Group + group merge is blocked with an explicit fix message; deleting a group is all-or-nothing; relative-bearing controls whether members re-orient with group turns. Grouping the **player's own** units must be done carefully since it forms a player-side action group.
- **Source:** vy5glbQ1G6k; wCaE47aRaHA
- **Confidence:** High

### Ship placement and air-complement assignment (by ship, not by group)
- **Models:** Placing surface groups and correctly basing their air complement.
- **Inputs / parameters:** **Insert** + click for ships; class filter; group via **G**; initial speed/heading; assign aircraft **"by ship" vs "by group."**
- **Behavior / rules:** Ships are placed like other units. When adding helicopters/aircraft to a surface action group you must add them **BY SHIP, not BY GROUP** — adding by group puts the air units in the **wrong places** (e.g. a Harrier ending up on a frigate). You can set the group's initial speed and heading at placement.
- **Outputs / effects:** Surface group with air complement correctly assigned to host vessels.
- **Edge cases / quirks:** By-group air assignment misplaces aircraft onto inappropriate hulls.
- **Source:** vy5glbQ1G6k
- **Confidence:** High

### Importing prebuilt unit packages (installations) & KML import
- **Models:** Loading author-supplied prebuilt facility/unit sets (real airfields, OOB packages) and converting Google-Earth KML points into units, instead of hand-building.
- **Inputs / parameters:**
  - **Import/Export Units:** `Editor → Import/Export Units → Load Groups/Installations`; country selection; "load selected installations."
  - **KML import:** a KML file (lat/long points); the community **"CMANO KML transfer"** tool (forums: search "command cmano kml transfer"); a target **database ID** to assign; the resulting Import/Export installation file.
- **Behavior / rules:**
  - **Prebuilt packages:** CMO ships libraries of prebuilt unit groups by type/country (e.g. several real Ecuadorian airports — including detailed piece-built ones; Syria ADW / Syria airbases / Russia-USSR packages). Imported items arrive **pre-GROUPED** (e.g. an air-defense package loads as one group with SA-2/SA-3/SA-6 SAMs plus EW). Selecting and loading drops them at their real locations. Workflow: import a country's bundle, then **prune/swap** individual elements (delete unwanted bases/units, replace detailed airbases with single-unit airfields when not targets). Make sure the correct **SIDE is active** before importing.
  - **KML → units:** (1) obtain a KML (e.g. "Syria chemical facilities KML"); (2) run the KML tool to convert KML → a Command-readable installation file; (3) the converted file lists each unit with a **placeholder DBID — the tool defaults them all to 10000**; (4) in a text editor, **find/replace 10000 with the desired real database ID** (example: industrial plant = **455**) for all entries — the file already carries the correct lat/long per point; also set the file's database-id field (set to **1** in the example); (5) drop the edited file into the correct side's Import/Export folder; (6) in-game `Import/Export → Load Groups/Installation`, select the side, and load — units appear precisely placed.
- **Outputs / effects:** Prebuilt, correctly-placed facilities/units, or many units instantiated at exact KML coordinates with a chosen platform type. Populated OOB then customized.
- **Edge cases / quirks:** Coverage is **incomplete** — imports are not available for every location on Earth. Placeholder DBID **10000 does nothing** and **MUST** be replaced. The file must be dropped under the **matching side's** import folder; DBID must match the intended platform or you get the wrong unit. Authors publishing their own imports should **GROUP** them ("much simpler to remove"). Watch anglicized place-name spelling when matching real bases. Beware **"scenario creep"** — start basic; each added unit multiplies effort. KML provides otherwise-missing valid conventional targets (e.g. chemical plants) that single-unit airfields can't supply.
- **Source:** vy5glbQ1G6k; ixu2x6doLFA; wCaE47aRaHA
- **Confidence:** High

### Air-unit quantity & availability modeling (squadron/regiment + one-third rule)
- **Models:** Deciding how many aircraft to place per base and how many are *operational at once*, based on real-world unit structure, readiness, spares, and maintenance.
- **Inputs / parameters:** Squadron size; regiment structure; number of hardened shelters/stands at the base; spare fraction; per-aircraft readiness state (Ready/armed, Arming, Maintenance/Not-available); national budget / aircraft maintainability (qualitative); aircraft-type maintenance difficulty (qualitative).
- **Behavior / rules:**
  - **Real-structure counts:** Western **squadrons** ~**12–24** aircraft (US fighter squadron **24**; F-15E squadrons smaller at **18**; carrier squadrons **12**). Russian/Soviet **regiments** are larger: nominally **three groups of ~15** (~**45**), but roughly **2/3 are spares** and unavailable, so a "full" regiment yields ~**36** aircraft with only ~**12** actually available at a time. Bomber regiments similarly ~**36**, not all usable.
  - **One-third operational rule (stated repeatedly):** about **one third** of a group's aircraft are operational at any moment (e.g. 24 F-18Es ⇒ **8** set to a mission, rest to maintenance). The designer manually assigns the subset: ~one third armed/ready, often another third "arming," remainder to maintenance/not-available.
  - **Availability adjusted by judgment:** well-funded/maintainable types (MiG-21, MiG-29 favored) get **higher** availability (e.g. "half," or 6 of a squadron); hard-to-maintain/older types (MiG-23, MiG-25, Su-24 Fencer) get **fewer** (often ≤4 — "if you have more than four ready I'm impressed" for Syria).
  - **Physical cap:** constrain counts by the number of **shelters/stands** at the base (e.g. 20 shelters ⇒ fewer aircraft; 30 stands ⇒ up to that many). Split a unit into sub-groups as needed (e.g. 6 CAS + 6 CAP).
- **Outputs / effects:** Realistic per-base aircraft counts, group splits, and how many are launchable at scenario start; fewer active units ⇒ less lag. A difficulty/performance lever.
- **Edge cases / quirks:** Numbers are **author heuristics, not hard CMO rules**. The author suggests literally **rolling dice** for availability; you may simply **DELETE** airframes you'll never use to save units, or set unwanted-flyable aircraft to **Maintenance** when role-playing. Some designers reject the one-third rule ("if the aircraft exists it's available"). The 2/3-spare rule and "three groups of 15" are explicitly approximations.
- **Source:** ixu2x6doLFA; b5RbZgiSpPU
- **Confidence:** Med

### Scenario clock micro-step to populate selectable forces ("run a millisecond, then reset")
- **Models:** A quirk where the engine only exposes/instantiates placed units to editor selection after the sim has ticked at least once.
- **Inputs / parameters:** `Scenario Times and Updates` (start time) menu; a momentary unpause.
- **Behavior / rules:** To make all placed units visible/selectable for equipping, run the scenario for a tiny instant ("a millisecond"), then return to `Scenario Times and Updates` and **RESET the start time**. Stated purpose: "so we're able to populate everything that we can see."
- **Outputs / effects:** All units become available to select and equip in the editor; scenario start time restored to original.
- **Edge cases / quirks:** Easy to forget the reset, which leaves the scenario started past T=0. Presented as a non-obvious authoring step ("it's going to seem a little weird").
- **Source:** b5RbZgiSpPU
- **Confidence:** Med

## 4. Airfield construction

### Single-unit (point) airfield vs multi-unit (piece-built) airfield
- **Models:** Two fundamentally different ways to represent an airbase, trading targetability for performance/simplicity.
- **Inputs / parameters:** Facility choice: a **single "airfield" facility** (specify runway lengths + counts; same single-unit format exists for ports) **vs** a hand-built **group of individual facilities** (runway, hangars, parking, fuel, ammo, radars, buildings).
- **Behavior / rules:**
  - **SINGLE-UNIT airfield:** inserted as **ONE** object (`Insert → Facility → "airfield" → choose runway lengths & counts → OK`); quick to populate; **CANNOT be attacked except by NUCLEAR weapons** (conventionally **invulnerable** even though it abstractly represents many structures); **low lag**. Recommended whenever the field is **not** meant to be a target. Hosted aircraft are **auto-distributed** across hangars/parking (visible via F6 air-facilities view). General rule of thumb: do not put **99+** units in a scenario when not needed — replacing non-target airbases with single-unit airports is the **primary performance lever** ("runs great," avoids lag).
  - **MULTI-UNIT (piece-built) airfield (required when it must be a conventional target):** place each component by hand; its pieces are **individually conventionally targetable**.
- **Outputs / effects:** Either a compact, conventionally-non-targetable point (low lag), or a detailed conventionally-attackable airfield complex.
- **Edge cases / quirks:** **Do NOT mix** a single-unit airfield with hand-built pieces ("it will melt on you"). **Do NOT build two separate multi-unit airports and group them together** — causes "really glitchy behavior." Trade-off: a single-unit airport cannot be partially destroyed building-by-building; granular ground-target destruction requires the full build-out and its performance cost. Two-runway airbases noted as a desirable pick. Hangar **facing direction does NOT affect anything functionally** (cosmetic).
- **Source:** vy5glbQ1G6k; 5xB8RNooK50; b5RbZgiSpPU
- **Confidence:** High

### Runway length measurement from map scale
- **Models:** Sizing a placed runway facility to match the real tarmac.
- **Inputs / parameters:** Measured runway length via the directional/measuring tool (nautical miles) and a heading; conversion **1 nm = 1852 m**.
- **Behavior / rules:** Use **terrain mode** to clearly see tarmac (recommended over satellite/Sentinel graphics). Measure the runway with the directional tool to get its length in nautical miles, then convert to meters by **multiplying by 1852** (worked: **1.7 nm × 1852 ≈ 3100 m**, rounded to **3200**). To place the runway **centered**, go to the **HALFWAY** point (half the length, e.g. **1600 m** for a ~3200 m runway) before pressing **Insert** and selecting the runway facility of the matching length (e.g. "3200 meter runway"). Second example: **1.2 nm ≈ 2400 m**, half = **1200 m**.
- **Outputs / effects:** A runway facility of the correct length placed at the correct spot.
- **Edge cases / quirks:** Choose the runway-length bucket nearest the computed meters.
- **Source:** 5xB8RNooK50
- **Confidence:** High

### Runway orientation uses TRUE heading; naming needs magnetic variation; orientation drives takeoff/landing direction
- **Models:** Aligning a runway with reality, labeling it correctly, and setting traffic-flow direction. (Also applies to an airfield's overall orientation = its landing-approach direction.)
- **Inputs / parameters:** **True** heading measured on the map; local **magnetic variation** (degrees); runway orientation set via right-click; (real wind direction as author judgment).
- **Behavior / rules:** Set the runway's orientation (`right-click → set orientation`) to match the measured **TRUE heading** — **CMO uses true, not magnetic, headings** (worked example: true heading **306/307°** applied directly). The runway **NUMBER/label**, however, is **magnetic**: to derive the designator you **ADD magnetic variation** to the true heading (example: variation **15°**, 307 + 15 ≈ 322 ⇒ likely **"runway 32,"** not "runway 31"). Orientation also **DICTATES the takeoff/landing direction**, so it must be correct for traffic flow; parallel runways/taxiways should share matching orientation. An **airfield's orientation** likewise defines the direction aircraft fly **TOWARD in order to LAND** — align it with the prevailing wind and avoid pointing the approach through terrain (e.g. landing through a mountain).
- **Outputs / effects:** Runway/airfield oriented for correct alignment, traffic flow, and landing path; correct designator if labeled.
- **Edge cases / quirks:** Confusing true vs magnetic leads to a mislabeled or misaligned runway; the actual designator could shift by a degree bucket — verify before labeling. A poorly chosen airfield orientation can route landings into terrain.
- **Source:** 5xB8RNooK50; vy5glbQ1G6k
- **Confidence:** High

### Runway-grade taxiways are usable as runways
- **Models:** Auxiliary paved surfaces that also function for flight ops.
- **Inputs / parameters:** Placement of a **"runway-grade taxiway"** facility (with length/orientation).
- **Behavior / rules:** A runway-grade taxiway can be **taken off from and landed on**, so it provides an **ADDITIONAL** set of operating surfaces/targets, unlike an ordinary taxiway. Treated and oriented like a runway (example ~**2600 m**, set parallel to the adjacent runway).
- **Outputs / effects:** An extra operable (and targetable) flight surface.
- **Edge cases / quirks:** A common authoring mistake — flagged as a **frequent cause of airfields "not working"** — is placing a **runway-grade taxiway when a regular runway was intended**.
- **Source:** 5xB8RNooK50
- **Confidence:** Med

### Access points connect runways to the airfield interior and gate attack difficulty
- **Models:** The internal road/route network that lets aircraft reach runways, and the redundancy that determines how hard the field is to disable.
- **Inputs / parameters:** **Access-point** facilities placed where movement routes meet runways/taxiways; access-point **size class** (e.g. "very large aircraft").
- **Behavior / rules:** An airfield **REQUIRES access points** or aircraft cannot reach the runway (a top cause of "broken" airfields is forgetting them). Place access points where ground routes intersect the runway/taxiway. **CRITICAL RULE:** as long as **ONE working access point remains**, an aircraft can hypothetically reach **anywhere** in the entire airfield. Therefore adding **many** access points (e.g. one per runway intersection) makes the field **HARDER to disable** (the attacker must destroy them all); concentrating **few** access points (example: ~**4**, at the main parking/apron connections) makes it **EASIER to shut down**. The author deliberately chooses a small set tied to main parking areas as the designed vulnerability.
- **Outputs / effects:** A connected, operable airfield whose resilience to attack scales with access-point count.
- **Edge cases / quirks:** Choose an access-point size class matching the largest aircraft using it ("very large aircraft"). Marking every runway access point yields an extremely resilient base — a **design lever, not an error**.
- **Source:** 5xB8RNooK50
- **Confidence:** High

### Facility orientation affects hit geometry (footprint must match building shape)
- **Models:** How a structure's modeled length/width and rotation determine the angle from which munitions strike it.
- **Inputs / parameters:** Facility orientation (rotation) and the facility's database **width/length** dimensions (e.g. aircraft hangar width **70**, length **120**).
- **Behavior / rules:** Placed facilities default to **north** orientation, which may not match the real building's footprint. Because a building has distinct width vs length (example hangar **70 × 120**), an incorrectly rotated (e.g. sideways) building presents the **wrong cross-section**, so bombing runs "hit it from the wrong angle." Orient hangars/buildings to match their **true footprint** for correct damage geometry.
- **Outputs / effects:** Correct strike geometry / vulnerability profile for the structure.
- **Edge cases / quirks:** Default-north placement is the trap; open the facility to read its width vs length **before** rotating.
- **Source:** 5xB8RNooK50
- **Confidence:** Med

### Ammo storage, fuel storage, and rearm dependency at airfields
- **Models:** Logistics structures that gate refuel/rearm and become destroyable choke points.
- **Inputs / parameters:** **Ammo storage** facility (loaded with specific weapons into its magazines); **fuel storage** facility; the **"Unlimited Magazines"** scenario-feature flag.
- **Behavior / rules:** **Load ammo storage with ammunition BEFORE grouping** the airfield (simpler and safer). With **Unlimited Magazines OFF**, aircraft can only rearm from weapons actually stocked in an ammo-storage facility's magazines; if a required weapon isn't stocked, that loadout is **unavailable** at Ready/Arm. To stock: select the ammo-storage unit, open **Magazines**, add weapons (filter by keyword or "show only weapons capable with aircraft"), and set quantities per weapon (examples: Mavericks/HARMs/Sidewinders/Sparrows at counts like 100/200/50). If the ammo-storage building is **destroyed**, aircraft can **no longer be rearmed** (a strategic vulnerability). Fuel storage similarly represents fuel supply and is a destructible vulnerability.
- **Outputs / effects:** Determines which loadouts are armable and ties rearm capability to a destroyable facility.
- **Edge cases / quirks:** With **Unlimited Magazines ON**, stocking is unnecessary and **all loadouts** are available. Weapons must be added per-type with explicit counts. Arming individual aircraft **after** regrouping causes them to be moved/relocated to rearm.
- **Source:** 5xB8RNooK50
- **Confidence:** High

### Placing aircraft into specific spots vs auto-placement on group-add
- **Models:** Where parked aircraft physically sit on the airfield and how that placement is controlled.
- **Inputs / parameters:** Adding aircraft to an **individual sub-facility** (select facility, **Ctrl+6** / Ctrl+F6) vs adding aircraft to the **whole airfield group**; the facility type (parking / hangar / alert "Christmas tree" apron).
- **Behavior / rules:** If you add aircraft to the **ENTIRE** airfield at once, CMO auto-assigns them to the "**most logical spot**" (often a hangar). To force aircraft onto a **specific** apron/parking facility (e.g. the alert "Christmas tree" so they're ready to scramble rather than "sitting out in the woods"), select that specific facility and add the aircraft to it (**Ctrl+6**) at the component stage — **before final grouping** — for best results.
- **Outputs / effects:** Aircraft parked at the intended specific locations on the field.
- **Edge cases / quirks:** **WARNING:** assigning a loadout (Ready/Arm) to those individually-placed aircraft **AFTER regrouping** causes them to be **moved around again** to be rearmed, defeating the placement.
- **Source:** 5xB8RNooK50
- **Confidence:** High

### Grouping an airfield into one unit and selecting the group lead
- **Models:** Consolidating many facilities into a single named airbase unit and choosing its representative icon.
- **Inputs / parameters:** **Box-select** all airfield facilities; **G** to group; **F4** to pick the lead unit; "Set Group Lead" on a chosen unit.
- **Behavior / rules:** Click-and-drag to select **every** component of the airfield (capture all detail), wait for it to register, then press **G** to group them into one unit. The lead/center may default to an arbitrary facility; to control which facility represents the group, click a unit and press **F4**, or select the desired unit and use **"Set Group Lead."** After grouping, the airfield acts as one operable base: launched aircraft taxi via access points to the main runway and take off; on recovery (**B** key) they re-enter via an access point and return to a parking spot.
- **Outputs / effects:** A single grouped airbase unit with a chosen lead, fully functional for launch/recovery.
- **Edge cases / quirks:** Must capture **ALL** facilities in the selection box. **Manual launch** is recommended for ballistic-weapon strikes against such fields because area saturation is imprecise (example strike only damaged one building).
- **Source:** 5xB8RNooK50
- **Confidence:** High

### Cargo loading on a facility/unit (e.g. barracks carrying infantry)
- **Models:** Assigning transportable cargo (personnel/equipment) to a unit for cargo/airlift missions.
- **Inputs / parameters:** `Edit Unit Actions → Edit Cargo`; cargo item type and quantity (example: **infantry section ×20**).
- **Behavior / rules:** A building such as a **barracks** can be designated and then loaded with cargo via the unit's `Edit Unit Actions → Cargo` menu (example loads an infantry section of 20). That cargo can then be picked up/transported for a cargo mission.
- **Outputs / effects:** The unit holds defined cargo available for transport missions.
- **Edge cases / quirks:** Presented as an advanced/optional flourish; barracks chosen as the cargo source in the example.

## 5. Loadouts, magazines & readiness

### Hosted-aircraft loadout, readiness states & time-to-ready
- **Models:** Stationing aircraft at a base and managing their alert/arming posture and which are combat-ready at T=0.
- **Inputs / parameters:** **Edit Hosted Aircraft** (map side filter, an era **"From" button** to show period-available types, quantity, callsign); per-group readiness: **Ready/Arm immediately, Maintenance, Reserve**; Ready/Arm **loadout** selection; explicit **Time-to-Ready** value; a **"ready immediately"** toggle.
- **Behavior / rules:** On a single-unit field you **Add** aircraft directly. The **"From"** button highlights which aircraft **TYPES** were available at the scenario's set **date** (e.g. Mirage 5 variants in '85). Quantity and callsign are set (callsign used to subdivide). In the map-editor you can group aircraft and set readiness **instantly**: **Ready** (armed and immediately available), **Maintenance**, or **Reserve**, choose loadouts via **Ready/Arm**, and even set an explicit **time-to-ready** (because it's the editor). **KEY behavior:** for aircraft to be usable at the very **START**, you must set them **"ready immediately" / pre-armed** — otherwise they are not available at scenario open and must arm first. Role is implied by the loadout (single-seat E-models → CAS/escort; F-models → precision strike; HARM loadouts → SEAD; air-to-air missiles → CAP/escort). Loadout count drives munitions math (e.g. 4 HARMs × 8 aircraft = 32 HARMs). The Ready/Arm "instant" buttons only appear if the base **has the weapons** (or Unlimited Ammo is on).
- **Outputs / effects:** Aircraft stationed at the base in chosen alert states with chosen loadouts and readiness timers; the starting airborne-capable weapons inventory at T=0; each aircraft's role.
- **Edge cases / quirks:** Arm-with-weapon options are **unavailable** without unlimited ammo or stocked magazines; the "From"/era filter keys off scenario **date**. Aircraft that return to base must **RE-ARM** before flying again. Loadout choice is a **difficulty lever** — hand players Mk-82 dumb bombs to make it hard, or PGMs to make it easy.
- **Source:** vy5glbQ1G6k; b5RbZgiSpPU
- **Confidence:** High

### Base magazines and weapon stocking
- **Models:** Local weapon inventory that gates which loadouts a base can arm and depletes as it arms aircraft.
- **Inputs / parameters:** `Magazines → Add Magazine` (choose a magazine **template**, e.g. a national-carrier magazine); **Add/Remove Weapons** within the magazine (the weapon must **match exactly** the one to be loaded, e.g. MK-82); resulting per-weapon counts.
- **Behavior / rules:** With **Unlimited Ammo OFF** you must add a magazine to a base before arming. Add a magazine template, then add/remove specific weapons inside it; the weapon you add must **EQUAL** the weapon you intend to load. Arming aircraft from the magazine **DECREMENTS** the stored count (e.g. 51 MK-82s reduces as bombs are loaded).
- **Outputs / effects:** A finite weapon stock at the base that constrains and is consumed by arming.
- **Edge cases / quirks:** There is **no generic "magazine #1"** template — you pick a named one and edit its contents; weapon identity must match exactly or the load won't apply.
- **Source:** vy5glbQ1G6k
- **Confidence:** High

### Rearm / turnaround timing as a planning constraint
- **Models:** Long re-arm/refuel times constrain when assets are available, shaping the operation timeline.
- **Inputs / parameters:** Per-loadout rearm time (e.g. Tu-22M swap to cruise missiles ≈ **20 hours**; Su-24/Su-17 rearm = hours; SAM reload ≈ **15 min**); aircraft availability windows; night-capability constraints.
- **Behavior / rules:** Plan around rearm times. Example: Tu-22Ms keep their FAB-500 loadout instead of swapping to cruise missiles because the swap costs ~**20 hours** and you want damage **now**. Strike packages become **unavailable** for their rearm duration after committing. SAM batteries are out ~**15 min** after expending a salvo. Some helicopters (certain Russian types) **CANNOT operate at night** and will not launch until daytime.
- **Outputs / effects:** A timeline of when each asset can (re)engage; constrains wave scheduling.
- **Edge cases / quirks:** Huge rearm penalties (20 h) make loadout swaps impractical mid-fight; night-only restrictions silently prevent some helo launches until daytime.
- **Source:** cMYv6wzbvmI
- **Confidence:** Med

### Designer awareness of player-exploitable weapon ranges/loadouts
- **Models:** Editing starting loadouts to preserve a scenario's intended challenge (preventing trivial long-range solutions).
- **Inputs / parameters:** Onboard weapons of player units (e.g. **Tomahawk TLAM** cruise missiles, **SM-2ER** long-range SAMs); platform strike radius vs target distance.
- **Behavior / rules:** After composing player forces, audit loadouts for capabilities that bypass the intended fight: e.g. **Tomahawks** (a "smart player will use them like crazy") and **SM-2ER** SAMs whose range is so wide they could down enemy aircraft from the sea ("eliminate all the fun of the air combat"). To preserve design, **REMOVE** such weapons from starting magazines if you don't want them used (or cap them, e.g. "reduce to 5" Tomahawks). Also size striker forces/tankers to route distance — worked distances: ~**550–600 nm** Black Sea transit; KH-22 reach **1475**; carrier kept **≥50 nm** off coast because SSMs likely operate within ~**50 nm** of the coast.
- **Outputs / effects:** A loadout/positioning configuration tuned to the intended difficulty.
- **Edge cases / quirks:** Pure design guidance, not an engine rule. The 50 nm coastal SSM standoff and ~600 nm strike radius are scenario-specific estimates. Long-range escorts may require adding **aerial refueling** (tankers) to reach the target area.
- **Source:** wCaE47aRaHA
- **Confidence:** Med

- **Outputs / effects:** No tanker units placed for that side.
- **Edge cases / quirks:** Contrast: long-range Su-27s in a companion video **ran out of fuel** mid-strike and had to refuel, which "doesn't work very well" — long-range missions need fuel planning or they fail.

## 7. Mission authoring

### Patrol mission authoring (BARCAP) with movement & flight-size controls
- **Models:** Defining an autonomous combat air patrol over reference geometry — drawn as a **line** of reference points rather than an area to stop wandering.
- **Inputs / parameters:** Select reference points + mission hotkey (**Ctrl+F11**); mission type (**BARCAP / Patrol AAW**); **Movement Style** (e.g. **Repeatable Loop**); **flight size** + the **"1/3 rule"**; a **"do not go exploring" / engage-outside-zone (leave-row-to-engage)** toggle; per-patrol engagement doctrine (e.g. **hit-and-run** = single attack then RTB); airframe with sufficient range (e.g. Su-33); per-mission **"allow refueling"** setting.
- **Behavior / rules:** Author selects reference points, invokes the mission editor (**Ctrl+F11**), picks **BARCAP/patrol**, and sets **Movement Style to a repeatable loop**.
  - Authoring a BARCAP as a **LINE** between reference points (not a patrol **AREA**) is preferred because area-patrol aircraft tend to **WANDER** out of the area and get shot down by SAMs (build-version-dependent behavior). A carrier/SAG CAP can stay an **AREA** because those aircraft stay close enough.
  - The **"1/3 rule"** keeps ~one third of assigned aircraft on station (with the rule, **4 of 12** stay airborne, in groups of four; "one section per shift").
  - The **engage-outside-zone / leave-row** toggle: **ON** makes aircraft leave the line to chase strikeable contacts (useful but risky — can send them into threats and get them shot down); **OFF** keeps them on the barrier.
  - Per-mission doctrine can be **hit-and-run** (one attack pass, then return).
  - **CRITICAL pitfalls:** (a) **NOT** setting **Repeatable Loop** causes them to fly off and not patrol; (b) leaving **"allow refueling" ON** makes patrol aircraft fly **ACROSS enemy SAMs** to reach a tanker — set refueling to **"not allowed"** for forward CAPs. Verify the assigned aircraft aren't already assigned to another mission.
- **Outputs / effects:** An autonomous patrol that cycles aircraft over the area/line per the movement/flight-size rules and auto-engages intruders.
- **Edge cases / quirks:** Aircraft with too little range immediately **bingo-fuel and RTB** over large countries. Area patrols wander into threats; missing Repeatable Loop breaks the loop; "allow refueling" on a forward CAP routes it suicidally across SAMs.
- **Source:** vy5glbQ1G6k; cMYv6wzbvmI
- **Confidence:** High

### Patrol/CAP mission timing relative to player launch
- **Models:** Choreographing AI aircraft to be airborne and on-station exactly when an attacking human player arrives, then standing down to save fuel.
- **Inputs / parameters:** Mission type (AAW/CAP patrol); a defined patrol **area** (e.g. via `Ctrl+right-click → Create Area → Rectangle`); units assigned; per-mission **station / transit / attack altitudes**; mission **activation (delay)** time and **deactivation** time; distance-to-area and aircraft transit speed.
- **Behavior / rules:** (1) Build a patrol area (e.g. constrained inside friendly Lebanon airspace, avoiding Turkey). (2) Assign aircraft, keeping ~**4 per class** airborne at a time (one-third logic applied to airborne presence; ~**20** airborne at once out of many assigned). (3) **Compute travel:** stated **100 nm ⇒ 15 minutes**; set activation **earlier** than that so jets are already up — example: scenario starts **03:30**, set activation **03:40** (wait 10 min, not the full 15), and **DEACTIVATE ~13:30** (~10 hours later) to stop wasting fuel. (4) Lower **station/transit altitude** deliberately (example **12,000 ft** station and **12,000 ft** transit) to stay below threats (NATO ship SAMs over a mountain range). Exploits that a "good player launches as soon as they unpause," so timing your own activation lets you meet them.
- **Outputs / effects:** AI fighters airborne, low, and on-station to intercept; mission auto-deactivates later to conserve fuel.
- **Edge cases / quirks:** If activated **too LATE**, the defending air force isn't present when the strike arrives (observed: Syrian AF "not that present," concluded activation should have been sooner). Altitude must be tuned to local SAM/terrain or friendly aircraft get shot down.
- **Source:** b5RbZgiSpPU
- **Confidence:** High

### Mission delay (activation) and deactivation windows
- **Models:** Time-gating when an AI mission becomes live and when it stands down.
- **Inputs / parameters:** Mission **activation** DTG; mission **deactivation** DTG; scenario start time.
- **Behavior / rules:** A mission can be given an activation time (does nothing until then) and a deactivation time (stops afterward). Used both to synchronize with enemy arrival and to stop fuel waste once the threat window passes. Concrete values: activate **03:40** (10 min after a 03:30 start), deactivate **~13:30**.
- **Outputs / effects:** Mission becomes active/inactive at the specified DTGs.
- **Edge cases / quirks:** Setting activation later than needed leaves defenders absent during the critical window.
- **Source:** b5RbZgiSpPU
- **Confidence:** High

### Inactive missions and staged activation for synchronized strikes
- **Models:** Holding a planned mission dormant, then activating it to launch on cue (H-hour).
- **Inputs / parameters:** Mission **active/inactive** flag; aircraft readiness; desired H-hour.
- **Behavior / rules:** Create missions as **INACTIVE** so their aircraft don't launch prematurely; when ready (confirmed SEAD effect, or a scheduled H-hour like **0700**), **ACTIVATE** — at which point **all ready assigned aircraft launch immediately**. If left active, ready aircraft launch as soon as they're armed, which **desynchronizes** a coordinated attack.
- **Outputs / effects:** Controlled, on-cue mass launch instead of trickle launches.
- **Edge cases / quirks:** An active mission **auto-launches ready aircraft** (breaks synchronization); remember to disable/inactivate to prevent premature relaunch after a first wave.
- **Source:** cMYv6wzbvmI
- **Confidence:** High

### Support mission (tanker + AEW track)
- **Models:** A patrol track that bases refueling tankers and AEW aircraft to sustain other CAPs.
- **Inputs / parameters:** Reference points; mission type **Support**; assigned tankers (Midas) and AEW (Mainstay); doctrine.
- **Behavior / rules:** Create a **Support** mission over reference points **behind the threat line**; assign refueling tankers and AEW aircraft. A support/tanker track is **required behind any barrier CAP** so CAP fighters can refuel **on station** instead of constantly cycling back to base.
- **Outputs / effects:** On-station refueling and airborne early warning sustaining forward patrols.

### Land / anti-runway strike mission planning (and the strike-planner altitude limitation)
- **Models:** Composing airbase strikes by aimpoint type, with per-mission minimum-attack-altitude control and correct munition-to-target pairing.
- **Inputs / parameters:** Mission type **Land Strike**; aimpoint selection by **category** (runways vs access points/taxiways vs hangars); assigned strike packages (anti-runway-bomb Su-25s for runways; MiG-29 / Su-25T for access points; Tu-22M "Backfire" for accuracy/bunkers); package size (e.g. 3-aircraft cells); **min attack altitude** (e.g. **>15,000 ft**); mission doctrine **offensive ECM on**; active/inactive flag.
- **Behavior / rules:** Separate a strike into pieces by what you're hitting: **anti-runway bombs** against **runways**, **FAB-500s** against **hangars**, etc. (set via `doctrine → weapons-to-release` so a patrol uses the right weapon per aimpoint). Use the **most accurate** platform for hard targets (Tu-22M for a command bunker). Keep packages **large** for maximum damage. Set a **MINIMUM attack altitude (>15,000 ft)** so strikers stay above short-range AAA/IR/ZSU threats. Enable **offensive ECM** in mission doctrine. Keep missions **INACTIVE** until the threat is suppressed, then activate (all ready aircraft launch on activation). **Cleanup discipline:** when done, set the mission inactive and **REMOVE units** from it rather than deleting it if it will be reused, then individually order those aircraft **RTB**. **KNOWN LIMITATION:** the strike planner ("strike mode") does **not** let you cleanly set the attack altitude per-aimpoint the way a **Patrol** mission's doctrine does — the recommended robust alternative is to define an **AREA + Land Patrol** and set the minimum altitude there.
- **Outputs / effects:** Coordinated multi-package airbase strike with altitude floors and correct munition-to-target pairing.
- **Edge cases / quirks:** Strike-planner can't enforce attack altitude as flexibly as a patrol — use area/patrol instead. Activating a mission launches **all ready aircraft at once**. Over-low attack altitude exposes strikers to AAA/ZSU/IR even after SAM suppression. Reusing a mission: inactivate + clear members rather than delete. The strike planner also delivers units **piecemeal / at different times** — add **dog-leg waypoints** to time-coordinate arrivals.

### SEAD "patrol-as-strike" missile-spam technique
- **Models:** Using a patrol-line SEAD mission with synchronized arrival and altitude steps to mass anti-radiation fire and saturate a SAM belt.
- **Inputs / parameters:** A short patrol **LINE (~25–30 nmi)** aimed into the target area; **all** ARM-carrying aircraft assigned; **flight size = MAXIMUM**; **1/3 rule OFF** (everyone airborne together); transit altitude (e.g. **36,000 ft**, above the ~20,000 ft cloud tops), on-station altitude **just under** cloud level, attack altitude (~**19,500–20,050 ft**); **Repeatable Loop**; doctrine investigate-unknown / weapons-range-contacts on.
- **Behavior / rules:** Rather than a defined area, draw a SEAD patrol as a **line** from point A to B aimed into the SAM belt and route **ALL** anti-radiation-missile aircraft along it. As successive flights arrive and shoot, then the next flights shoot, producing **missile spam** that saturates all the SAM radars. Set flight size to **MAX** and turn **OFF** the 1/3 rule so **every** ARM shooter is airborne simultaneously (quantity matters because modern SAMs can shoot down ARMs). **Altitude scheme:** TRANSIT high (**36,000 ft**, above clouds) to avoid ZSU-23 (capped at **10,000 ft AGL**) and use cloud cover; on station **dip just under the clouds** so SAM radars cue and launch; aircraft then **pop UP into the cloud and fire ARMs back** — radars expose themselves to launch and get hit. After firing, send them home, **re-angle the line**, and repeat. **Synchronize arrival:** if some aircraft arrive first they get killed piecemeal, so add extra **legs/doglegs** to the route to **delay early arrivals** so all hit the target window together. To clear ~**25 SAM sites** you need on the order of **200 ARMs** (25 sites × ~4 each + extras) — i.e. repeat the spam several times.
- **Outputs / effects:** Mass simultaneous ARM launches that suppress an entire SAM belt; radars that expose themselves get destroyed.
- **Edge cases / quirks:** Piecemeal/early arrivals get killed — add route legs to synchronize. ARM aircraft told to "attack any target" waste ARMs on **early-warning radars en route**. Insufficient ARM quantity only suppresses a fraction (~quarter) per pass. SEAD aircraft also wander if not looped.

- **Confidence:** High

## 8. Defensive laydown design (radar siting, EW networks, SAMs, IADS)

### Radar siting governed by terrain line-of-sight and Earth-curvature/altitude
- **Models:** Where ground radars can actually see, limited by terrain masking and the radar horizon.
- **Inputs / parameters:** Radar position (lat/long + terrain elevation at that point); radar type & range; target test altitude; the **Line-of-Sight (LOS)** tool with a **refresh** and a **target-altitude** input.
- **Behavior / rules:** Use the **LOS tool**: set a target altitude (e.g. **3000 m ≈ 10,000 ft**) and **refresh** to see what the radar covers. **Terrain (mountains) MASKS coverage** — a radar in a valley/low spot is blind beyond surrounding peaks even at 10,000 ft target altitude. The **radar HORIZON grows with the radar's own altitude** (Earth curvature hides things behind the bulge), so siting a radar on the **HIGHEST nearby terrain** extends coverage. Author workflow: hover terrain to read elevation, hunt the tallest peak, place there, refresh LOS, **nudge position while watching LOS** until blind spots close. For a water-surrounded island, push radars to the **farthest/most-westerly peninsulas/peaks** to see as far seaward as possible. Practical limit: the LOS tool has a **max range cap**.
- **Outputs / effects:** Effective radar coverage footprint; blind spots behind terrain and below the horizon.
- **Edge cases / quirks:** Cranking radar range with LOS active can **crash/melt the tool** ("melt OBS"); a radar's coverage is only as good as its **terrain siting** regardless of nominal range. Example real ranges quoted: long-track ~**200 nmi**, P-18 ~**200 nmi**, Squat-Eye ~**110 nmi**.
- **Source:** sfxAYvnk8FM
- **Confidence:** High

### Early-warning radar network design (layered EW coverage)
- **Models:** Building a multi-radar early-warning net so no approach axis is unwatched.
- **Inputs / parameters:** Number/placement of EW radars; terrain; threat approach axes (over-water gaps, island channels); radars must be **turned ON**.
- **Behavior / rules:** Author philosophy: place "**as many EW radars as it takes**" to leave no uncovered approach. Identify water gaps the enemy must cross (e.g. ~**12 nmi** through the Scottish Isles gives little warning) and cover them; combine **long-range area radars** in the interior (sited high) with **peninsula radars facing seaward**. **CRITICAL operational note:** radars do nothing if you don't **TURN THEM ON**.
- **Outputs / effects:** Overlapping detection coverage of all ingress routes.
- **Edge cases / quirks:** Short over-water gaps inherently give little warning; an un-activated radar provides **zero** coverage (explicitly demonstrated as a mistake).
- **Source:** sfxAYvnk8FM
- **Confidence:** High

### SAM placement by system class/role (area vs point, high vs low altitude)
- **Models:** Matching each SAM type's range/speed/altitude envelope and role to terrain and target clusters.
- **Inputs / parameters:** SAM system class — **SA-2** (area), **SA-3** (faster/shorter, for leakers), **SA-4** (long-range, **few missiles**), **SA-5** (very long range, area), **SA-6** (fast point), **SA-8/OSA** (very short ~**5–10 nmi**, very accurate, anti-PGM), **SA-9** (very short IR, ambush); terrain (peaks vs valleys, forest cover); target clusters to defend.
- **Behavior / rules:** Doctrine taught: deploy **area SAMs (SA-2/SA-5)** to umbrella clusters of strategic targets from **high terrain** overlooking approach valleys; use **shorter, faster SAMs (SA-3/SA-6)** to catch **"leakers"** that beat the area SAMs; place very-short-range accurate systems (**SA-8/OSA**) **right beside point targets** (even on the hill next to a power plant) specifically to shoot down incoming **PRECISION-GUIDED MUNITIONS** at the asset; place very-short-range IR systems (**SA-9**) in **low-altitude ambush spots** where the enemy is expected to sneak through (in the **gaps between** existing SAM coverage). **Low-altitude systems go IN valleys** to trap low flyers, **not on peaks**. **Fixed SAMs (SA-2)** are known to the enemy and will be targeted, so use **forest/terrain cover**. Guiding maxim: **"If you try to protect everything you protect nothing."**
- **Outputs / effects:** A layered SAM laydown mapping each system's envelope to the right target/role/terrain.
- **Edge cases / quirks:** **SA-4** has long range but **few missiles** (runs dry fast); putting low-altitude systems on **peaks** wastes them on high flyers; gaps between area-SAM coverage are exactly where **IR ambush SAMs** belong; engagement-reaction time can be too slow if a system is placed in a **deep valley** vs on the adjacent hill.
- **Source:** sfxAYvnk8FM
- **Confidence:** High

### IADS integration via inter-unit communications (and selective de-integration)
- **Models:** Whether SAM/radar units share targeting data as one integrated network.
- **Inputs / parameters:** Default: **all units of a side can communicate**; `right-click → set a SAM "out of communication"`; optional Lua to auto-enable SAM radars when EW radars are lost.
- **Behavior / rules:** By default every unit on a side can talk to every other, so **EW radars automatically pass targeting info to SAMs** — this **IS** the integrated air-defense system. To simulate a **NON-integrated/degraded** system, **right-click a SAM and set it out of communication** so it cannot receive/share targeting data. Advanced: a Lua command linked to a trigger can **auto-activate SAM-site radars if the EW radars are destroyed**.
- **Outputs / effects:** Controls whether SAMs benefit from networked cueing or must fight blind/independently.
- **Edge cases / quirks:** De-integrating a SAM **sharply reduces its efficiency**; the auto-radar-on-EW-loss behavior requires **Lua + a trigger** (outside basic scope).
- **Source:** sfxAYvnk8FM
- **Confidence:** High

### Weapons Release Authority (WRA) per target type and salvo sizing
- **Models:** Per-platform rules for which target categories to engage, how many rounds, and at what range. (Also applies generally to limiting how many of a munition a unit expends per engagement.)
- **Inputs / parameters:** Per-target-type entries (**Unspecified, Fifth-gen fighter, High-perf bomber, Reconnaissance, AEW/AWACS, Helicopter, Guided weapon, UAV**, etc.); **rounds-per-engagement** count; **automatic firing range** limit. Also configurable per weapon type (e.g. **HARM, ALARM** → "fire ONE, no more").
- **Behavior / rules:** Set, for each target category, whether to engage and how many rounds. Guidance: do **NOT** fire at targets you can't identify ("unspecified" off) or you **draw anti-radiation missiles**; against high-performance bombers use **3 rounds** because bombers carry defensive ECM + chaff (a single shot is ~**55% Pk** per missile but ECM/chaff degrade it); do **NOT** waste missiles on helicopters or on incoming guided weapons (you'll be destroyed as soon as you fire a salvo anyway). **KEY assumption:** a fixed SAM typically gets only **ONE salvo** before it is destroyed by ARMs — so make it count. Disable engaging fighters with bomber-killer SAMs (SA-3) to conserve ammo for bombers. Set the **AUTOMATIC FIRING RANGE inside the no-escape zone, NOT at max range** — firing at max range loses line-of-sight before impact. Quoted: SA-2 set to ~**20 nmi** (within DLZ); SA-5 max ~**100 nmi** but ~**40 nmi ≈ 3,000 ft** / ~**100 nmi ≈ 15,000 ft** engageable target altitude (so don't fire at max range); SA-9 fire **2** because per-missile Pk is very low. For ARM-carriers generally, set HARM release to **fire one, no more** so AI doesn't squander anti-radiation missiles.
- **Outputs / effects:** Governs what each platform engages, salvo depth, and the range band it opens fire in; AI fires at most the specified number per engagement.
- **Edge cases / quirks:** Max-range fire breaks LOS before impact; the one-salvo-before-death assumption drives **3-round** salvos on bombers; ammo-limited systems (SA-5 with ~**60 rockets**, SA-6 with only **6**) need rationing; WRA can be re-tuned mid-scenario once threats (e.g. ARM stocks) change. Over-firing is a real observed problem the author actively corrects.
- **Source:** sfxAYvnk8FM; 0SwTlMuRdzo
- **Confidence:** High

### Identification gating before engagement (visual aids; cloud effects)
- **Models:** You must classify a contact before you should fire, and ID is helped by sensors/conditions.
- **Inputs / parameters:** Forward observers / **watchtower** facilities with **binoculars** and/or **night-vision (2nd-gen)** sensors; cloud cover; day/night; the SA-5's ability to **ID a target by its radar emission**.
- **Behavior / rules:** Do not authorize firing at **unidentified** contacts (draws ARMs and wastes missiles). To improve ID, add a forward-observer group such as a **watchtower** with **binoculars** (visual ID of seen targets) and **night-vision (2nd-gen)** for night attacks. **Heavy cloud and night degrade identification.** Some systems (**SA-5**) can identify a target by its **emitting radar**, enabling engagement without visual ID.
- **Outputs / effects:** Higher-confidence contact classification enabling discriminate engagement.
- **Edge cases / quirks:** The watchtower **spotlight** sensor is "not that useful"; clouds/night make ID harder; SA-5 emission-ID is a special capability.
- **Source:** sfxAYvnk8FM
- **Confidence:** Med

### Radar emission vs survivability tradeoff (rotation flicker, ARM lethality)
- **Models:** Turning on fire-control radar exposes a SAM to anti-radiation attack.
- **Inputs / parameters:** Radar on/off state; rotating search-radar behavior; presence of enemy ARMs (Shrike/Martel/HARM-class).
- **Behavior / rules:** A **rotating search radar** shows contacts **intermittently** (appears/disappears as the beam sweeps). The **moment a fire-control radar lights up**, the SAM becomes an ARM target and will likely be destroyed (**"lifespan ~30 seconds"**). Therefore only illuminate/engage targets actually **worth the risk**; expect ~**15 minutes** to **reload** a SAM after expending its salvo.
- **Outputs / effects:** Detection of contacts vs immediate vulnerability to anti-radiation weapons.
- **Edge cases / quirks:** Search-radar flicker is normal (beam rotation); ARMs and SAMs can race — sometimes the SAM kills the shooter before its own ARM arrives; reload latency (~15 min) sidelines a battery after one salvo.
- **Source:** sfxAYvnk8FM
- **Confidence:** High

### Modern phased-array EW radars and stealth detection
- **Models:** Newer, shorter-range but stealth-capable phased-array radars replacing long-range legacy EW.
- **Inputs / parameters:** Phased-array radar type; placement altitude/LOS (pre-tuned); target stealth/RCS; radar wavelength vs target size.
- **Behavior / rules:** Modern phased-array radars have **LESS range** than legacy systems but are **tremendously better at detecting stealth** targets. They are placed at good altitudes with checked LOS just like legacy EW. **Detection physics note:** a smaller wavelength makes a **BIGGER** object easier to detect — so a very large stealth platform (**B-2**) is **harder** to detect than smaller stealth aircraft (explaining why "only one B-2 was caught").
- **Outputs / effects:** Reduced-range but stealth-capable early warning.
- **Edge cases / quirks:** Range/stealth tradeoff vs legacy; counterintuitive size-vs-wavelength detection (large stealth = harder).
- **Source:** flw8O10fxPE
- **Confidence:** Med

### Modern SAM laydown: long-range area (S-300/S-400) + point-defense anti-PGM
- **Models:** Modern integrated defense built around very-long-range area SAMs plus dedicated PGM-killers.
- **Inputs / parameters:** **S-400** ("SA-21," long range, many simultaneous shots); **S-300** variants (**PT/P/PT-1/PM** "modernized" / **PMU** "very nice" / **PS** "ranged"; **SA-10** fixed/strategic vs **SA-12** mobile, developed in parallel); point defense: **SA-15** (short-range, accurate), **SA-19**, **Pantsir** ("SA-22," excellent anti-PGM); **search-radar enable** toggle; **battalion-radar enable** toggle.
- **Behavior / rules:** Place the longest-range area SAM (**S-400**) on the key **peninsula** to deny a whole flank, launching at aircraft **as they leave the ground**. Distribute **S-300 variants** on **tall peaks** to cover sectors. Use **SA-15/SA-19/SA-22** as **point defense behind** the area SAMs, assigning the **better anti-PGM system (SA-22/Pantsir)** to the **most important assets**. **SA-10 = fixed/strategic**, **SA-12 = mobile**. Scandalous-but-necessary in the modern context: **ENABLE the search/battalion radars** even though it exposes them, because the **ONLY way to detect incoming precision-guided munitions early is by RADAR** (vision/IR is too late at night or in cloud).
- **Outputs / effects:** A layered modern IADS optimized to intercept both shooters and their standoff munitions.
- **Edge cases / quirks:** Enabling search/battalion radars exposes batteries but is **required to see PGMs**; S-400 "wastes" shots at long range on aircraft but is satisfying vs stealth; the designation "SA-22" vs "SA-24" for Pantsir is **uncertain in the narration**.
- **Source:** flw8O10fxPE
- **Confidence:** High

### Jammers vs PGMs / camera feeds / GPS

### Exclusion Zone for automatic hostile classification at borders

## 9. The Event Editor (triggers / conditions / actions)

### Event editor structure: trigger + condition + action
- **Models:** CMO's scripted-event system for scoring, messages, and scenario-state changes — the no-Lua way to script outcomes. `Editor → Event Editor → Events`.
- **Inputs / parameters:** Event **name**; **Repeatable** flag; optional **probability**; one or more **Triggers**; zero or more **Conditions**; one or more **Actions**.
- **Behavior / rules:** Each event has **THREE** parts evaluated in order: **Trigger** (fires/activates the event), **Condition** (a gate/catch that must also be true), **Action** (what happens). An event = (a Trigger fires) **AND** (all Conditions true) → run Actions. Create a new event, **NAME** it, optionally mark it **Repeatable**, optionally set a **probability**. You can attach **MULTIPLE** triggers (e.g. "merchant destroyed" OR "airplane destroyed") and **MULTIPLE** actions. Triggers/conditions/actions are **reusable libraries** — create once, attach to many events (cloning an existing one is the fast path). **Pitfalls called out:** after creating a trigger/condition/action you must explicitly click **Add** to attach it (the #1 mistake — "make sure you actually add it"); and forgetting **Repeatable** makes a per-loss event fire only once.
- **Outputs / effects:** A live event that, when its trigger fires and conditions pass, runs its actions.
- **Edge cases / quirks:** Created sub-elements must be explicitly **ADDED** to the event; non-repeatable events fire once; probability can randomize firing. The Event Editor is explicitly **distinct from and weaker than Lua/SBR** — the "potent but not scripting" tier.
- **Source:** TCApvEgog4U; 0SwTlMuRdzo
- **Confidence:** High

### Trigger types (Unit Destroyed, Side Points threshold) with target specificity
- **Models:** The conditions that activate an event.
- **Inputs / parameters:**
  - **Unit Is Destroyed:** target **Side**, target **Type** (Surface Ship / Aircraft / Submarine / Facility…), optional **subtype** (e.g. Merchant), optional specific **Class**, optional specific **Unit**.
  - **Side Points:** target **Side** + a points **threshold**.
- **Behavior / rules:** **"Unit Is Destroyed"** can be scoped from broad (any aircraft of a side: type Aircraft, any/any/any) down to a very specific exact class or exact hull — the selectors are **hierarchical** (side → type → subtype → class → unit). **"Side Points"** fires when a side's score **exceeds** a value — entered as **THRESHOLD-MINUS-ONE**: to fire at **30** points you set **29**, because the trigger means "**greater than 29**."
- **Outputs / effects:** Activates the owning event when the described destruction or score condition occurs.
- **Edge cases / quirks:** The points threshold is **strictly "greater than the entered number,"** so enter target-minus-one. (CMO also has additional trigger types — time, unit-enters-area, etc. — that were **not demonstrated** in the read transcripts; see Part 2 / the live editor.)
- **Source:** TCApvEgog4U; 0SwTlMuRdzo
- **Confidence:** High

### Event conditions (the three demonstrated)
- **Models:** Boolean gates layered on top of triggers.
- **Inputs / parameters:** Three condition kinds: **side-relationship** (is hostile / is friendly), **"Scenario Has Started,"** and a **special-script (Lua)** condition.
- **Behavior / rules:** The tutorial states there are essentially **three** condition types. A near-universal one is **"Scenario Has Started"** (so scoring/effects only apply after start — created once as a reusable condition and reused on every event, preventing events from firing during setup/initial spawn). Conditions are **ANDed** with the trigger.
- **Outputs / effects:** Restricts whether a triggered event's actions execute.
- **Edge cases / quirks:** The **"special script"** condition is the Lua escape hatch (out of scope / blocked for RMOOZ).
- **Source:** TCApvEgog4U
- **Confidence:** High

### Event actions (Points, End Scenario, Message, Teleport, Mission Active/Inactive, Lua)
- **Models:** What an event does when it fires.
- **Inputs / parameters:** **Points** action (side + signed value, e.g. +15, or −10 entered as a negative); **End Scenario** action; **Message** action (side + text); **Teleport to area**; **Change Mission active/inactive**; **Lua script** action; ("add planes"/spawn referenced as available).
- **Behavior / rules:** **Points** adds (or subtracts, via a negative value) points to a chosen side — the core scoring primitive. **End Scenario** terminates the scenario (typically **non-repeatable**, since it's over). **Message** posts text to a side's message log (or any side). **Teleport** moves units to an area (enables a randomized "button" that could teleport one side across the map). **Mission active/inactive** flips a mission on/off. **Lua** action available but out of scope. Multiple actions per event run together; pair a Points action with a Message action for player feedback.
- **Outputs / effects:** Mutates score, ends the scenario, posts log messages, relocates/spawns units, or toggles missions.
- **Edge cases / quirks:** Negative points are entered as a **negative number**; End-Scenario events need not be repeatable; teleport can be combined with **probability** for randomized effects.
- **Source:** TCApvEgog4U; 0SwTlMuRdzo
- **Confidence:** High

### Score/loss logging and scoring graph
- **Models:** Automatic tally of points and attrition driven by events.
- **Inputs / parameters:** Event **point actions**; unit losses.
- **Behavior / rules:** As point-actions fire, the side **score updates** and is recorded in a **scoring log** and a **scoring graph** (rises/falls over time). A **losses** tally records units expended/destroyed per side. Test events by **deliberately triggering** them and checking the Scoring / scoring-log / scoring-graph.
- **Outputs / effects:** Running score, score-vs-time graph, and per-side loss/expenditure records.
- **Edge cases / quirks:** None stated. (Canonical worked patterns: "merchant destroyed" → +15 + message, repeatable; "lose airplane" → −10, repeatable; "win" → Side Points >29 → End Scenario, **not** repeatable — each guarded by "scenario has started" and cloned from a base trigger/action.)
- **Source:** TCApvEgog4U; 0SwTlMuRdzo

## 10. Testing, balancing & in-game strike allocation

### Iterative test–rebalance loop with versioned saves
- **Models:** Author workflow — run the scenario, observe losses/difficulty/fun, tweak, save a new version, repeat.
- **Inputs / parameters:** Test runs (unpaused, time-accelerated); **Losses & Expenditures** screen; **Scoring** screen; subjective "am I having fun / is it too easy/hard" judgment.
- **Behavior / rules:** Repeatedly build a "fake airstrike," run it, study interactions/losses/difficulty, then edit loadouts/timings/units. Strong discipline: **press Ctrl+S after essentially every change** ("every time you twitch your hand"), and **VERSION** scenarios (save-as O2, etc.) so builds are recoverable. During testing the designer may grab/move units by hand or force-declare contacts hostile purely to exercise scoring.
- **Outputs / effects:** Progressively balanced scenario; multiple saved versions.
- **Edge cases / quirks:** Strong warning against **accidental saves** overwriting a known-good state ("try not to save by accident" when reloading). **"Scenario creep"** is the named anti-pattern — adding too much makes it laggy and unfun.
- **Source:** 0SwTlMuRdzo
- **Confidence:** High

### Loadout reduction as a balance / difficulty lever
- **Models:** Tuning strike lethality by changing how many weapons each platform carries.
- **Inputs / parameters:** Per-aircraft weapon **loadout quantity** (e.g. cruise-missile count per Tu-22).
- **Behavior / rules:** After testing showed **5 of 9** chemical plants destroyed by a Tu-22 cruise-missile salvo, the designer reopened the scenario and set Tu-22s to carry a **SINGLE** cruise missile each — a "very subtle change" that **"dramatically" reduces** effectiveness, leaving the player something to do. General principle: shrink loadouts to rebalance an over-powerful AI strike; conversely cap player weapons (e.g. "reduce to 5" Tomahawks) to force the hard way.
- **Outputs / effects:** Lower per-platform damage output; a more balanced scenario.
- **Edge cases / quirks:** Long-range standoff cruise missiles (e.g. **JASSM**) can "**win the scenario automatically**" — if you want a sandbox, leave them; for challenge, remove/limit them (recommends removing B-1s entirely as they trivialize it).
- **Source:** 0SwTlMuRdzo
- **Confidence:** High

### Auto-Strike planner (engine auto-allocates weapons to targets)
- **Models:** A one-click strike order where the engine predicts how many weapons are needed and launches them — with a manual override.
- **Inputs / parameters:** A target or set of targets; available standoff weapons (e.g. Tomahawks); a manual allocation override (N per target).
- **Behavior / rules:** Order a unit to "**do a strike**" on a target (or "all chemical facilities") and the engine "**automatically predicts exactly how many they need**" and launches (e.g. auto Tomahawks). Alternatively the player **manually allocates** N per target (examples: "allocate 27" / 5 each = **45** Tomahawks; or precise "allocate **2**" per B-1). Manual allocation lets you be deliberately cheap or wasteful.
- **Outputs / effects:** Weapons launched against targets, either auto-sized or at the allocated count.
- **Edge cases / quirks:** Auto-allocation tends to **OVER-spend** (27/45 Tomahawks "wasted" to kill few plants); a manual two-missile B-1 allocation killed **8 of 9** targets far more cheaply — illustrating **cost-efficiency** as a hidden scenario objective.
- **Source:** 0SwTlMuRdzo
- **Confidence:** High

## 11. Quick Battle generator (data-driven)

### Quick Battle definition via HTML + Lua + stylesheet triplet
- **Models:** Each Quick Battle scenario template is a self-contained mini-engagement generator stored as **editable files** (not a bespoke UI).
- **Inputs / parameters:** Folder name (becomes the battle display name); an **HTML** file (dropdown option lists), a **Lua** script (random-table logic + generation), a **CSS** stylesheet (presentation).
- **Behavior / rules:** Each Quick Battle option on the menu is backed by a folder containing **one HTML, one Lua, one stylesheet**. The **FOLDER NAME dictates the battle name** shown in the menu (rename the folder ⇒ rename the battle). The HTML defines the selectable dropdown options; the Lua backs it with logic and generates the **random table** of values. HTML **feeds values into the Lua**, which generates battles dynamically. Files live under the install path, e.g. Steam: `steamapps/common/Command Modern Operations/QuickBattle/<battle folder>`. Edit HTML/Lua with a **plain text editor (Notepad)**; do **NOT double-click** the HTML (opens rendered, not editable).
- **Outputs / effects:** A selectable Quick Battle whose options, units, and random pools are determined by the edited files.
- **Edge cases / quirks:** **Copy the entire battle folder as a backup** before editing and rename the copy. Command reports an error if you "goofed up" the file. You can build an entirely custom Lua mission and use HTML only to feed values in.
- **Source:** Qc4pohpgRmU
- **Confidence:** High

### Unit encoding in Quick Battle HTML: DBID_loadoutID (and carried-unit chaining)
- **Models:** How a selectable platform-plus-weapons option is represented as raw database identifiers.
- **Inputs / parameters:** Database ID of the platform (unit); database ID of the **loadout**; optional database ID of a **carried sub-unit** (e.g. embarked helicopter) which itself has its **own loadout**.
- **Behavior / rules:** Each dropdown option value references database entries by **numeric ID**. The format pairs a **unit with its loadout**; the underscore **"_"** separates entries. Example: an **FFH** (helicopter frigate) is DBID **959**; its helicopter (MH-60) is **3202**; the helicopter's loadout (ASW patrol) is **2206**. To change a carried helicopter to e.g. a **Helix**, swap the unit DBID (e.g. **4327**) **AND** supply that unit's matching loadout DBID — **every loadout has its own database ID**. A loadout ID is **MANDATORY** for every unit entry. To find IDs, open the **in-game database viewer** and read the entry number for the platform and each loadout.
- **Outputs / effects:** The Quick Battle spawns the specified platform carrying the specified loadout (and embarked unit with its loadout).
- **Edge cases / quirks:** A unit **without** a loadout "will freak out" / error. The pattern is always **DatabaseID followed by loadoutID**. Use the **MODERN database** (not classic) for these example battles — option pools differ between databases. (Transcript spoken IDs are contiguous, e.g. 959, 3202, 2206, 4327, 507.)
- **Source:** Qc4pohpgRmU
- **Confidence:** High

### Random-option pool lives in Lua, not HTML
- **Models:** Whether a newly-added unit is eligible to appear when the player picks the "random" selection.
- **Inputs / parameters:** A **random table** of values defined in the Lua script (list of DBID,loadoutID pairs).
- **Behavior / rules:** The **"random"** selection for a slot is generated from a **random table in the Lua script, NOT from the HTML** option list. Adding a new unit only to the HTML makes it **manually selectable** but does **NOT** include it in random draws. To include a unit in the random pool you must also add its database ID and loadout ID into the **Lua random table** (e.g. append **"507, 507"** for a unit whose DBID and loadout are both 507).
- **Outputs / effects:** The unit becomes eligible to be drawn when "random" is chosen.
- **Edge cases / quirks:** **Save the Lua file** or edits are lost. Changing which **side** units belong to is possible in the Lua but risky — a mistake there "can accidentally break something." Beginners should edit only values they're comfortable with.
- **Source:** Qc4pohpgRmU
- **Confidence:** High

### Quick Battle location model: center lat/long + random-radius placement
- **Models:** How the two combatants are geographically positioned for a generated Quick Battle.
- **Inputs / parameters:** A named game location (HTML variable name + human description), **latitude** (decimal degrees), **longitude** (decimal degrees), and a **GMT offset**; an internally-computed random radius.
- **Behavior / rules:** A location is defined by a single **CENTER** longitude/latitude point. CMO then extrapolates a **radius from that center RANDOMLY** to place the **OTHER** unit (one unit at/near center, the second at a random offset within the radius). Adding a custom location requires editing **BOTH** files: (1) HTML "game location" section — copy an existing example, set the internal **variable name (no spaces, no fancy symbols, or the HTML breaks)** and a human-readable description; (2) Lua "game locations" section — add a matching entry keyed by the **same variable name** carrying latitude, longitude, and the GMT-offset value.
- **Outputs / effects:** A selectable location; on generation, units are placed around that center, the second at a random bearing/radius.
- **Edge cases / quirks:** In the Lua locations list, **every entry EXCEPT the last must end with a comma; the last must NOT** — a missing comma "blows up" parsing (treats entries as one line). The description string is **cosmetic only** (not shown in HTML). Same technique works for air-to-air or any engagement type.
- **Source:** SKyOl8umX6s
- **Confidence:** High

### GMT offset derivation for Quick Battle time-of-day
- **Models:** Keeps a chosen time-of-day (e.g. 12:00) meaningful at the battle's real-world location.
- **Inputs / parameters:** Zulu (UTC) time and local time at the desired battle location.
- **Behavior / rules:** Because "12 o'clock isn't the same time everywhere," each location stores a **GMT offset** so a set time-of-day maps to correct local lighting. To compute it, compare **Zulu time to local time** at the target point; the difference is the offset (worked example: local is **4 hours ahead** of Zulu ⇒ offset = **4**). Enter that integer as the location's GMT offset in the Lua.
- **Outputs / effects:** Time-of-day for the generated battle resolves to the correct local solar time at that location.
- **Edge cases / quirks:** Sign convention follows hours ahead/behind UTC; the example used **positive 4**. Treated as the simplest of the location parameters.
- **Source:** SKyOl8umX6s
- **Confidence:** Med

### Coordinate format conversion (DMS in UI → decimal degrees in script) with hemisphere signing
- **Models:** Translating the map UI's coordinate readout into the numeric form the Quick Battle script consumes.
- **Inputs / parameters:** Latitude and longitude as **degrees-minutes-seconds** (read off the cursor on the map), output as **signed decimal degrees**.
- **Behavior / rules:** Command's on-screen cursor gives coordinates in **degrees/minutes/seconds**, but the Lua script expects **DECIMAL degrees**, so you must convert (the author uses an external "degrees minutes seconds to decimal degrees" converter such as latlong.net). Then apply **hemisphere signs:** latitude **POSITIVE for Northern, NEGATIVE for Southern**; longitude **POSITIVE for Eastern, NEGATIVE for Western**. Paste the latitude into the latitude field and longitude into the longitude field of the location entry.
- **Outputs / effects:** A correctly placed center coordinate for the location.
- **Edge cases / quirks:** Omitting the sign places units in the wrong hemisphere (author's quip: people end up "in the middle of the Great Lakes"). Easy to **swap latitude/longitude** — explicitly warned to keep them straight.
- **Source:** SKyOl8umX6s
- **Confidence:** High
